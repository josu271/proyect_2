# 2\. Riesgos e Incidentes (Risks and Issues)**

Registro de los principales riesgos e incidentes que ocurrieron durante el proyecto, junto con las respuestas aplicadas y las lecciones para mejorar el aprendizaje organizacional.

| **Descripción del Riesgo / Incidente**                                                                           | **Respuesta Aplicada y Efectividad**                                                                                                                                                              | **Comentarios / Lección Aprendida**                                                                                                                  |
| ---------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| Complejidad del motor CSP con OR-Tools superó la estimación inicial de tiempo de implementación.                 | Se realizaron sprints de investigación técnica dedicados exclusivamente al motor. Se simplificaron algunas restricciones blandas para priorizar la entrega de las restricciones duras.            | Incluir spikes técnicos al inicio del proyecto para estimar con mayor precisión el esfuerzo de componentes de alta incertidumbre técnica.            |
| Conflictos de integración entre los módulos de backend y frontend en los puntos de API.                          | Se definió un contrato de API (endpoints, payloads) antes de la implementación paralela. Los conflictos se resolvieron en sesiones de pair programming entre el desarrollador backend y frontend. | Generar documentación automática de la API (OpenAPI/Swagger) desde el inicio del proyecto para reducir ambigüedades en los contratos de integración. |
| Prueba E2E multiusuario presentó fallos intermitentes por condiciones de carrera en la base de datos de pruebas. | Se configuró un ambiente de pruebas con datos de seeding reproducible. El test multiusuario fue ajustado con tiempos de espera explícitos (cy.wait) para estabilizar el flujo.                    | Implementar transacciones de base de datos con rollback automático entre tests para garantizar el aislamiento total de las pruebas E2E.              |

# 3\. Defectos de Calidad (Quality Defects)

Defectos identificados durante el proceso de pruebas, su resolución y las recomendaciones para proyectos futuros.

| **Descripción del Defecto**                                                                                                             | **Resolución**                                                                                                                                                                        | **Comentarios / Mejora Futura**                                                                                                          |
| --------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| El módulo de matrícula no validaba correctamente el límite de 6 créditos cuando el estudiante seleccionaba cursos de forma concurrente. | Se implementó validación tanto en el frontend (antes de enviar la solicitud) como en el backend (como restricción de negocio en el servicio de matrícula), garantizando consistencia. | Establecer doble validación frontend-backend como estándar de desarrollo desde el inicio, especialmente para reglas de negocio críticas. |
| La visualización del horario semanal no se adaptaba correctamente a resoluciones de pantalla menores a 768px (responsive design).       | Se refactorizó el componente de horario usando clases responsivas de Tailwind CSS y se validó con breakpoints estándar (sm, md, lg).                                                  | Incorporar pruebas de responsividad en el plan de QA para validar los componentes visuales más complejos en diferentes resoluciones.     |
| El token JWT no se invalidaba correctamente al cerrar sesión, permitiendo su reutilización durante el tiempo restante de expiración.    | Se implementó una lista negra de tokens (blacklist) en memoria para el contexto académico. Se documentó la limitación y se propuso una solución con Redis para producción.            | Diseñar el mecanismo de invalidación de tokens desde la arquitectura inicial, no como corrección posterior.                              |

# 4\. Gestión de Proveedores (Vendor Management)

El proyecto utilizó exclusivamente herramientas y librerías de código abierto. No se gestionaron proveedores externos de servicios. A continuación se registran las tecnologías de terceros utilizadas y su evaluación.

| **Componente**              | **Evaluación**                                                                                            | **Comentarios**                                                                                                                                          |
| --------------------------- | --------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| React + Vite (Frontend SPA) | Excelente. Ecosistema maduro, documentación completa, comunidad activa.                                   | Se recomienda continuar usando React para proyectos similares. La combinación con Vite mejoró significativamente el tiempo de build.                     |
| FastAPI (Backend REST)      | Excelente. Alta performance, documentación automática OpenAPI, tipo seguro con Pydantic.                  | FastAPI demostró ser una elección acertada para APIs REST en Python. Se recomienda para futuros proyectos de backend.                                    |
| OR-Tools (Motor CSP)        | Bueno. Potente para problemas de optimización combinatoria, aunque con curva de aprendizaje pronunciada.  | Se recomienda incluir un spike técnico de al menos 1 semana para familiarizarse con la API de OR-Tools antes de iniciar la implementación del motor CSP. |
| PostgreSQL (Base de datos)  | Excelente. Confiable, con soporte avanzado para consultas complejas necesarias para el motor de horarios. | Para producción, se recomienda configurar connection pooling (PgBouncer) para optimizar el rendimiento bajo carga concurrente.                           |

# 6\. Conclusión General

El equipo del proyecto SGDHA logró entregar un PMV funcional que cumple el objetivo general definido en el Project Charter, dentro del presupuesto programado y con desviaciones de cronograma menores y absorbidas en la fase de cierre. Las principales lecciones para futuros proyectos del equipo son: (1) definir criterios de aceptación detallados desde el primer sprint, (2) planificar las pruebas unitarias de forma distribuida en todos los sprints en lugar de concentrarse al final, y (3) mantener actualizada la documentación de revisión y retrospectiva al cierre de cada sprint, no solo al final del proyecto.
