| Planificación y control de calidad | Se ejecutaron pruebas unitarias, de integración, E2E (Cypress) y de seguridad (OWASP) en cada sprint; el 100% de los 12 defectos detectados fueron corregidos. | La cobertura de pruebas unitarias (objetivo ≥70%) no se alcanzó dentro del Sprint 4 (52%); se debió planificar la historia de pruebas (SGDHA-15) más temprano en el proyecto en lugar de dejarla para el último sprint. |
| Planificación de recursos físicos | El uso de equipos personales, GitHub y servicios cloud gratuitos/económicos (Railway, Vercel, Render) fue suficiente para todo el ciclo de desarrollo. | No se identificaron mejoras relevantes en esta área. |
| Planificación, desarrollo y desempeño del equipo | El equipo de 4 integrantes mantuvo roles claros (Backend, Frontend, QA/Documentación, Datos) y buena capacidad de respuesta ante bloqueos (ej. corrección del modelo de datos en 1 día). | Se identificó la necesidad de mejorar la coordinación entre integrantes y distribuir mejor la carga de trabajo desde el inicio de cada sprint (retrospectiva Sprint 1). |
| Gestión de comunicaciones | El equipo mantuvo Daily Scrum recurrentes (hasta 3 veces por semana en el Sprint 2\) y comunicación activa con usuarios representados (docente y estudiante). | En sprints iniciales las actualizaciones eran cada 2 semanas, un ritmo identificado como inadecuado; se recomienda Daily Scrum desde el primer sprint. |
| Reportes | Se generaron Informes de Estado del Proyecto para los 4 sprints, con métricas de alcance, cronograma, costos, calidad y riesgos. | Los reportes de revisión y retrospectiva de los sprints 2 a 4 se mantuvieron como plantilla sin actualizar durante la ejecución; se debieron completar inmediatamente al cierre de cada sprint. |
| Gestión de riesgos | Los riesgos técnicos de mayor impacto (motor CSP, autenticación JWT) fueron anticipados y mitigados con spikes técnicos antes de afectar el cronograma. | El riesgo de cobertura de pruebas insuficiente se identificó tarde (Sprint 4); un registro de riesgos más vivo y revisado en cada sprint habría anticipado mejor este punto. |
| Planificación y gestión de adquisiciones | No se gestionaron proveedores externos; las herramientas utilizadas (GitHub, Jira, servicios cloud) tuvieron planes gratuitos suficientes para el alcance académico del proyecto. | No aplica para este proyecto. |
| Participación de los interesados (stakeholders) | Se mantuvo retroalimentación directa con usuarios representados (docente y estudiante) para validar restricciones académicas reales desde el Sprint 1\. | Se puede formalizar más la participación del sponsor/evaluador en revisiones intermedias, no solo en la entrega final. |
| Información de mejora de procesos | La adopción de revisión de avance a mitad de sprint (desde el Sprint 3\) mejoró la distribución del trabajo en el tiempo respecto a los sprints anteriores. | Formalizar esta práctica desde el inicio del proyecto en futuros ciclos, en lugar de adoptarla como acción correctiva a mitad de proyecto. |
| Información específica del producto | El motor CSP alcanzó un tiempo de generación de 32s (objetivo ≤50s) y la optimización de carga redujo el tiempo de visualización en 75% (3.2s → 0.8s). | Explorar en una futura iteración la paralelización del algoritmo CSP para escenarios con mayor volumen de estudiantes y cursos. |

Conclusión General

El equipo del proyecto SGDHA logró entregar un PMV funcional que cumple el objetivo general definido en el Project Charter, dentro del presupuesto programado y con desviaciones de cronograma menores y absorbidas en la fase de cierre. Las principales lecciones para futuros proyectos del equipo son:  (1) definir criterios de aceptación detallados desde el primer sprint, (2) planificar las pruebas unitarias de forma distribuida en todos los sprints en lugar de concentrarse al final, y (3) mantener actualizada la documentación de revisión y retrospectiva al cierre de cada sprint, no solo al final del proyecto.
